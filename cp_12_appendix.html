Appendix B — macOS 原生命令速查与常见陷阱

这一章不是正文延伸，而是一份方便回查的附录。

前面的章节已经解释了为什么要这样用、什么时候该这样用；这里要做的事更简单：

当你已经知道问题大概属于哪一类时，能快速想起该先敲什么命令。

B.1 文件与目录速查

B.1.1 看清当前位置与目录内容

pwd
ls
ls -l
ls -la
ls -lh
ls -lt

最常见的作用：

  • pwd：确认你现在到底在哪个目录
  • ls：看当前目录下有什么
  • ls -l：看权限、大小、时间
  • ls -la：把隐藏文件也一起看出来
  • ls -lh：文件大小更容易读
  • ls -lt：先看最近改过什么

常见误区：

  • ls 看到的只是“当前目录”，不是整个项目
  • 默认看不到隐藏文件，所以 .zshrc、.git 之类经常被误判为“不存在”
  • 文件系统默认大小写不敏感，不代表远程 Linux 也这样

B.1.2 创建、复制、移动、删除

touch demo.txt
mkdir notes
cp source.txt backup.txt
cp -R srcdir backupdir
mv old.txt new.txt
rm file.txt
rm -r olddir

最常见的作用：

  • touch：创建空文件，或更新时间戳
  • mkdir：创建目录
  • cp：复制文件
  • cp -R：复制目录
  • mv：移动，或顺便改名
  • rm：删除文件
  • rm -r：递归删除目录

常见误区：

  • mv 不只是“移动”，经常也在“重命名”
  • rm 删除后默认不会进废纸篓
  • rm -r 一旦路径看错，破坏范围会很大

B.1.3 看空间与文件信息

du -h .
du -h . | sort -h
df -h
file archive.tar.gz
stat report.txt

最常见的作用：

  • du -h .：看当前目录里谁最占空间
  • du -h . | sort -h：从小到大排一下，异常对象更明显
  • df -h：看磁盘整体是不是快满了
  • file：确认文件到底是什么类型
  • stat：看更细的元数据

常见误区：

  • df -h 回答的是“磁盘整体”，不是“当前目录谁占空间”
  • du 关心的是目录和文件实际占用，不是 Finder 里看起来的名字
  • file 读的是文件内容特征，不是扩展名猜测

B.2 文本处理速查

B.2.1 先筛，再看全文

grep error server.log
grep -i error server.log
grep -v DEBUG server.log
grep -R TODO .
less server.log
head -n 20 server.log
tail -n 50 server.log

最常见的作用：

  • grep：找包含某个模式的行
  • grep -i：忽略大小写
  • grep -v：排除干扰信息
  • grep -R：递归搜索目录
  • less：分页查看长文件
  • head：只看开头
  • tail：只看结尾

常见误区：

  • grep 找的是“行”，不是“单词对象”
  • grep -R 在大目录里可能很慢，也可能扫到你不想扫的内容
  • less 是查看器，不会改文件

B.2.2 轻量替换与结构化处理

sed 's/error/ERROR/' server.log
sed -i '' 's/error/ERROR/' server.log
cut -d',' -f1 data.csv
sort names.txt
uniq names.txt
sort names.txt | uniq
wc -l server.log
awk '{print $1}' access.log

最常见的作用：

  • sed：做流式替换
  • sed -i ''：在 macOS 上原地修改文件
  • cut：按分隔符切字段
  • sort：排序
  • uniq：去掉相邻重复行
  • wc -l：统计行数
  • awk：按列提取内容

常见误区：

  • 在 macOS 上，sed -i 不能直接照抄很多 Linux 教程的写法
  • uniq 只能消除相邻重复，所以通常先 sort
  • awk 默认按空白分列，不是自动识别 CSV

B.3 查找与定位速查

B.3.1 找文件、找目录、找最近改过的东西

find . -name "*.log"
find . -iname "*.pdf"
find . -type d -name ".git"
find . -mtime -1
find . -size +100M
which python3
type ls
whereis ssh

最常见的作用：

  • find：递归定位文件或目录
  • -iname：忽略大小写匹配
  • -mtime -1：最近一天修改过
  • -size +100M：找体积异常的大文件
  • which：看命令会从哪里执行
  • type：判断它是外部命令、alias、函数还是 shell builtin
  • whereis：粗略看相关路径

常见误区：

  • which 不能完整解释所有 shell 行为，尤其是 alias / function
  • find 的 *.log 需要加引号，否则可能先被 shell 展开
  • “找不到文件”不一定是没有，也可能是你起点目录选错了

B.4 进程与系统状态速查

B.4.1 看谁在跑、谁最忙、谁该停

ps
ps -u $(whoami)
ps aux
top
kill 12345
kill -9 12345
killall Safari
uptime
whoami
id

最常见的作用：

  • ps：看进程快照
  • ps -u $(whoami)：只看自己启动的程序
  • ps aux：看更完整的进程信息
  • top：看当前谁最占 CPU / 内存
  • kill：请求一个进程退出
  • kill -9：强制终止
  • killall：按进程名结束
  • uptime：看系统运行时长和平均负载
  • id：确认当前用户和组

常见误区：

  • ps 不是实时监控，它只是瞬时快照
  • kill 不等于“暴力杀死”，默认先发的是可被处理的信号
  • kill -9 应该放在最后，而不是第一反应
  • killall 是按名字匹配，不是按窗口匹配

B.5 网络与远程速查

B.5.1 先判断“通不通”，再判断“哪里不通”

ping baidu.com
ping -c 4 google.com
ifconfig
netstat -an
curl https://example.com
curl -I https://example.com
ssh user@host
scp local.txt user@host:/tmp/

最常见的作用：

  • ping：确认基本连通性
  • ifconfig：看网卡状态和 IP
  • netstat -an：看连接和监听端口
  • curl：直接打一个 HTTP 请求
  • curl -I：只看响应头
  • ssh：远程登录
  • scp：远程拷贝文件

常见误区：

  • ping 不通，不一定说明网站“坏了”，也可能是对方禁了 ICMP
  • curl 成功，说明的是接口能访问，不等于页面逻辑没问题
  • ifconfig 更适合看状态，不是现代 Linux 教程里那种“统一网络管理入口”

B.6 macOS 独有高频命令速查

B.6.1 GUI 与 CLI 的桥梁

open .
open report.pdf
open -a "Visual Studio Code" .
open https://www.apple.com
pbcopy
pbpaste
say "hello"

最常见的作用：

  • open .：在 Finder 中打开当前目录
  • open file：交给系统默认应用处理
  • open -a：指定某个应用打开
  • open URL：在默认浏览器中打开链接
  • pbcopy：把命令输出放进剪贴板
  • pbpaste：把剪贴板内容送回终端
  • say：调用系统语音合成

常见误区：

  • open 调起的是 GUI 应用，不代表终端在“读取文件内容”
  • pbcopy 没有屏幕输出，不代表它没工作
  • 某些应用即使被 open -a 启动，也不一定接受你传入的文件或目录参数

B.7 macOS 与 Linux 最容易混淆的地方

这一节不是全面比较，而是只列最常踩坑的点。

B.7.1 sed 的原地修改写法不同

在很多 Linux 教程里你会看到：

sed -i 's/a/b/' file.txt

但在 macOS 上，常见写法是：

sed -i '' 's/a/b/' file.txt

如果你把 GNU sed 的例子原样抄过来，常常会直接报错。

B.7.2 stat 的参数风格不同

Linux 和 macOS 都有 stat，但选项和输出格式并不一致。

这意味着：

  • 交互式查看时问题不大
  • 一旦写成脚本，就必须确认当前系统是哪一种实现

B.7.3 默认工具是 BSD userland，不是 GNU userland

这会影响很多细节：

  • 选项名字不完全一样
  • 某些扩展功能不存在
  • 文档里“理所当然”的写法不一定适用

所以抄命令时，先问自己一句：

这是通用 Unix 写法，还是 GNU/Linux 特定写法？

B.8 高风险命令提醒

以下命令不是不能用，而是必须先确认路径、对象和范围：

rm
rm -r
mv
cp -R
sed -i ''
chmod
chown
kill -9

使用前建议先做的检查：

  1. pwd，确认当前位置
  2. ls，确认目标确实在这里
  3. 尽量先做“只查看、不修改”的版本
  4. 批量操作前，先拿一个样本验证

一个非常实用的习惯是：

当命令会“改东西”时，先写出“只看结果”的版本；当命令会“删东西”时，再多看一眼路径。

B.9 看到报错时，先查什么

B.9.1 command not found

先检查：

echo $PATH
which commandname
type commandname

你要确认的是：

  • 命令真的存在吗
  • 它在不在 PATH 里
  • 你调用的是可执行文件，还是 alias / function

B.9.2 No such file or directory

先检查：

pwd
ls
ls -la

你要确认的是：

  • 当前目录是不是你以为的那个目录
  • 文件名有没有写错
  • 隐藏文件是不是被你忽略了

B.9.3 Permission denied

先检查：

ls -l target
id

你要确认的是：

  • 你有没有执行权限 / 读取权限 / 写入权限
  • 这个文件是不是本来就不该由当前用户直接操作

B.9.4 改完文件没生效

先检查：

cat file
source ~/.zshrc
echo $SHELL

你要确认的是：

  • 你到底改的是哪个文件
  • 这是 shell 配置问题，还是应用自己的配置问题
  • 当前会话有没有重新加载

B.10 最后的使用原则

附录的价值，不在于让你背更多命令，而在于减少“第一步就走错”的概率。

真正高效的命令行使用方式通常都很朴素：

  • 先确认位置
  • 再确认对象
  • 先看，再改
  • 先局部验证，再批量操作
  • 不把 Linux 教程默认当成 macOS 现实

如果你能稳定做到这几件事，那么原生命令行就已经足够解决大量真实问题。
