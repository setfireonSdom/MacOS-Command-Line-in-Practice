4  查找与定位

在真实工作中，你要找的东西往往不是“这个名字的文件在哪里”。而是最近改过的那个配置文件、占空间异常的文件、不知道从哪来的可执行程序、Finder 明明能搜到，但我想在命令行里处理的那批文件。

这一章只用系统自带工具，回答一个问题：

macOS 在“找东西”这件事上，本来就给了你哪些能力？

4.1 find（原生 find 的能力边界）

find 是一个极其朴素、但可靠的工具。它的工作方式只有一句话：

从某个目录开始，递归遍历每一个文件，用规则逐个判断。

这也决定了它的能力边界：精确、可组合、不快（但稳定）、不“聪明”（但可预期）

4.1.1 案例 1：按名称查找（最基础，但最常用）

你想在当前目录下，找所有 .log 文件：

find . -name "*.log"

注意这里的几个关键点：

  • .：查找起点

  • -name：按文件名匹配

  • “*.log”：glob 写法，但由 find 自己解释

这是和 ls *.log 的一个重要区别，glob 是 Shell 展开，-name "*.log" 是 find 在判断。

4.1.2 案例 2：忽略大小写的查找

在 macOS 默认文件系统下，这个非常常见：

find . -iname "*.pdf"

你不需要猜文件到底是 README、Readme 还是 readme。

4.1.3 案例 3：按时间查找（现实世界非常常用）

你想找 最近一天内修改过的文件：

find . -mtime -1

含义是：

  • -mtime：按“修改时间”

  • -1：小于 1 天

如果你想找 7 天前的文件：

find . -mtime +7

这类用法在排查“最近谁动过什么”时非常有用。

4.1.4 案例 4：按大小查找（清理磁盘的第一步）

你想找大于 100MB 的文件：

find . -size +100M

这一步不是为了立刻删除，而是为了定位异常对象。

4.2 which / type

这一节解决的是一个非常具体、但经常困扰人的问题：

我现在用的这个命令，到底来自哪里？

4.2.1 案例 1：which —— 找“可执行文件在哪”

你输入：

which python

得到一个路径，比如：

/usr/bin/python

这意味着当你敲 python 时， Shell 实际执行的是这个文件。

这在以下场景中非常关键同名命令行为不一致、系统命令 vs 用户安装命令、PATH 顺序引发的问题。

4.2.2 案例 2：type —— 比 which 更诚实

type ls

你可能会看到：

ls is /bin/ls

这就是 type 的价值：

它告诉你，这个名字现在到底代表什么。

是别名、是内建命令、还是磁盘上的可执行文件。

这是 which 做不到的。

当行为和预期不一致时， 第一件事，用 type。

4.3 mdls / mdfind（Spotlight 的 CLI 接口）

这一节，是 macOS 原生命令行里最容易被低估的一部分。

Finder 搜索 ≠ 魔法，Finder 用的是 Spotlight 索引，mdfind 是它的命令行接口。

也就是说，你在 Finder 里能搜到的东西， 理论上在命令行里也能搜到。

4.3.1 案例 1：用 mdfind 直接搜索

你想找所有包含 “invoice” 的文件：

mdfind error

这一步不关心文件在哪、文件名是什么、后缀是什么，它依赖的是系统已经建立好的索引。

它的意思是在整个系统中搜索文件名或内容包含 invoice 的文件，包括文档、PDF、邮件、备忘录等所有被 Spotlight 索引的内容。输出文件的完整路径。

如果你要找的内容有引号，就这样：

mdfind "1,Alice,87"
mdfind "error 123"

引号内被视为一个整体。

4.3.2 案例 2：限制搜索范围

你只想在某个目录下搜：

mdfind -onlyin ~/Documents invoice

这在大型目录结构中非常实用。

4.3.3 案例 3：查看文件的元数据（mdls）

你对一个文件感兴趣，但想知道：创建时间、内容类型、Spotlight 认定的属性。

mdls test.txt

你会看到一组系统维护的元信息。这不是“文件内容”， 而是 系统对文件的理解结果。
